# pgsleuth-pg-ext

Postgres extension that exposes a small set of helper SQL functions used
by the userspace eBPF loader (`pgsleuth-ebpf-loader`) to bootstrap its
filters at startup.

The extension is part of the **v0 skeleton** described in
`pgsleuth/docs/research/Database Observability Alarms.md` (Skeleton POC
section). Its job is intentionally small — it proves the
loader ↔ extension handshake so later alarms can reuse the wiring.

## v0 SQL surface

| Function | Returns | Purpose |
| --- | --- | --- |
| `pgsleuth_wal_device()` | `TEXT` (`"major:minor"`) | Device ID of the filesystem holding `$PGDATA/pg_wal`. The loader uses this to filter eBPF block-layer events to the WAL device only (primary signal for fsync-jitter detection). |
| `pgsleuth_postmaster_pid()` | `INT4` | Postmaster PID, exposed so the loader can scope process-based filters without guessing. |

## Prerequisites

The build is **Docker-only** for now — the host does not need Postgres
headers or `cargo-pgrx`. Inside the `rust-dev` container (see
`pgsleuth/infra/docker/rust-dev.Dockerfile`):

| Tool | Version | Notes |
| --- | --- | --- |
| Rust toolchain | stable | Same as the rest of the workspace. |
| `cargo-pgrx` | `0.12.9` | Must match the `pgrx` version pinned in `Cargo.toml`. Pinned because pgrx and cargo-pgrx are version-locked. |
| `postgresql-server-dev-17` | apt (from `apt.postgresql.org`) | Provides `pg_config` + server headers. **See note below — this does *not* install the Postgres server.** |

### Note on `postgresql-server-dev-17`

The package name is misleading: despite "server" in the name, this is
the **headers-only** package. It does *not* install the Postgres
daemon, `initdb`, `psql`, a data directory, or any running-server
component.

What it actually provides:

- `pg_config` — the binary `pgrx` shells out to during build to discover
  include paths, lib paths, and configure flags.
- Server-side C headers under `/usr/include/postgresql/17/server/`
  (`postgres.h`, `fmgr.h`, `nodes/*.h`, the full extension ABI) —
  consumed by `pgrx`'s `bindgen` step.
- `libpq-dev` (transitive dependency) — `libpq-fe.h` for client-side
  connection code.
- `pgxs.mk` — Makefile snippets used by the standard extension build
  system (pgrx replaces these but they're harmless).

Installed footprint: ~30–50 MB. No background processes, no listening
sockets, no daemon scaffolding.

We can't go lighter:

- `libpq-dev` alone gives only client headers (`libpq-fe.h`) —
  insufficient for compiling a server-side extension.
- Debian / pgdg does not split server headers into a smaller subpackage;
  `postgresql-server-dev-17` *is* the minimum for building extensions
  against PG 17.

Why the official `apt.postgresql.org` repo and not Debian's default
mirrors: Bookworm's default repos only carry `postgresql-server-dev-15`.
The pgdg repo is the standard source for newer PG versions on Debian
and the same one the official `postgres:17-bookworm` image draws from
for the runtime container.

The actual `psql` used to install and smoke-test the extension lives in
the **`ebpf-feasibility` container** (which runs the full
`postgres:17-bookworm` image). Build (rust-dev) and runtime
(ebpf-feasibility) stay cleanly separated; rust-dev never runs a
Postgres server.

The `rust-dev` Dockerfile is the right place to install both
`cargo-pgrx` and `postgresql-server-dev-17`.

## Build

All commands are run inside the `rust-dev` container, from the
workspace root (`/workspace/source/pgsleuth-ebpf-poc/`):

```bash
# Build the extension and stage installable artifacts under ./target/.
cargo pgrx package --package pgsleuth-pg-ext --pg-config "$(which pg_config)"
```

`cargo pgrx package` produces a deployable layout in
`target/pgsleuth-pg-ext/pkg/usr/lib/postgresql/17/lib/pgsleuth.so` and
the matching `.control` + `.sql` files under
`target/pgsleuth-pg-ext/pkg/usr/share/postgresql/17/extension/`.

`cargo pgrx package` requires `$PGRX_HOME` (`~/.pgrx/`) to exist with a
config that points at a `pg_config`. The `rust-dev` Dockerfile handles
this once at image build time:

```dockerfile
RUN cargo pgrx init --pg17 "$(which pg_config)"
```

Pointing `--pg17` at the **system** `pg_config` means pgrx does **not**
download or build a managed Postgres — it just writes a config that
resolves `PG_CONFIG_PATH` to the system one. Takes seconds, not the
~30 min that the default `cargo pgrx init` (which builds PG 12–17 from
source) would.

## Install (into the `ebpf-feasibility` container)

After `cargo pgrx package`, copy the staged tree into the runtime
container's Postgres paths:

```bash
# From the rust-dev container, with /workspace/build mounted into
# ebpf-feasibility at the same path:
cp -r target/pgsleuth-pg-ext/pkg/usr/lib/postgresql/17/lib/pgsleuth.so \
      "$(pg_config --pkglibdir)/"
cp    pgsleuth.control \
      "$(pg_config --sharedir)/extension/"
cp    target/pgsleuth-pg-ext/pkg/usr/share/postgresql/17/extension/pgsleuth--*.sql \
      "$(pg_config --sharedir)/extension/"
```

In the runtime container, enable the extension once Postgres is up:

```bash
psql -U postgres -c "CREATE EXTENSION pgsleuth;"
```

The follow-up commit that wires this into
`pgsleuth/infra/docker/setup-postgres.sh` will run the `CREATE
EXTENSION` automatically at container startup.

## Verify

```sql
-- Should return a string like "259:0" (NVMe) or "8:1" (SATA).
SELECT pgsleuth_wal_device();

-- Should return a positive integer matching `pgrep -x postgres | head -1`.
SELECT pgsleuth_postmaster_pid();
```

## Local development (inside `rust-dev`)

The Dockerfile already runs `cargo pgrx init --pg17 "$(which pg_config)"`
at image-build time, which is enough for `cargo pgrx package`. For the
interactive `run` / `test` workflows below, pgrx will additionally
download + build a managed Postgres on first invocation — that's the
~30 min `init` step you want to avoid in CI but is fine for local dev.

```bash
# Iterate: build, install into the managed PG, start it, and open psql.
cargo pgrx run pg17 --package pgsleuth-pg-ext

# Run the embedded #[pg_test] suite against a managed PG.
cargo pgrx test pg17 --package pgsleuth-pg-ext
```

The managed PG instances live under `~/.pgrx/` inside the container and
are entirely separate from the `ebpf-feasibility` runtime PG.

## Build integration (cross-references)

This crate scaffolds the extension. The integration commits that follow
will:

1. Extend `pgsleuth/infra/docker/rust-dev.Dockerfile` with `cargo-pgrx`
   + `postgresql-server-dev-17`.
2. Extend `pgsleuth/infra/docker/ebpf-feasibility.Dockerfile` to copy
   the packaged artifacts into the right paths.
3. Modify `pgsleuth/infra/docker/setup-postgres.sh` to run
   `CREATE EXTENSION pgsleuth;` after Postgres starts.
4. Extend `pgsleuth-ebpf-loader` to query both functions at startup via
   `tokio-postgres` and seed its filter configuration.

Each of those is a separate, reviewable change. Track them in the
project board's backlog under **[T1] Alarm 03** (skeleton scope).
